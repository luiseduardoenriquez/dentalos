# TP-09: Generate Treatment Plan PDF Spec

---

## Overview

**Feature:** Generate a professional PDF document for a treatment plan, including clinic branding, patient demographics, CIE-10 diagnoses, full procedure list with costs, odontogram snapshot, total estimated cost, and approval signature (when plan is approved). Document is in Spanish and follows A4 clinical layout.

**Domain:** treatment-plans

**Priority:** High

**Dependencies:** TP-02 (plan-get.md), TP-08 (plan-approve.md), P-01 (patient-create.md), I-01 (multi-tenancy.md), Tenant branding settings

---

## Authentication

- **Level:** Privileged
- **Roles allowed:** doctor, clinic_owner, assistant
- **Tenant context:** Required — resolved from JWT
- **Special rules:** Patient role may access their own plan's PDF via a temporary signed link (generated by TP-10 share endpoint). Direct API access requires staff role.

---

## Endpoint

```
GET /api/v1/patients/{patient_id}/treatment-plans/{plan_id}/pdf
```

**Rate Limiting:**
- 20 requests per minute per user (PDF generation is CPU-intensive)
- Burst allowance: 5 concurrent generation requests per tenant

## Business Logic

**Step-by-step process:**

1. Validate URL and query parameters.
2. Resolve tenant from JWT claims; set `search_path` to tenant schema.
3. Check user permissions via RBAC (must be doctor, clinic_owner, or assistant).
4. Fetch plan and verify it exists and belongs to the patient.
5. If `regenerate = false`, check cache/storage: query `plan_pdfs` table for an existing PDF generated after the plan's last `updated_at`. If cached PDF exists and is not stale, return its pre-signed URL without regeneration.
6. If generation required: fetch all required data in parallel:
   a. Full plan data (TP-02 logic: items, diagnoses, approval info).
   b. Patient demographics (name, document, birthdate, gender, city).
   c. Responsible doctor name and credentials.
   d. Tenant branding (clinic name, logo URL, address, phone, NIT/tax ID).
   e. Odontogram snapshot image (PNG from odontogram service for current state).
   f. Approval signature image from S3 (if plan is approved).
7. Render HTML template (`treatment_plan_es.html`) with Jinja2, substituting all fetched data.
   - Language: Spanish (es-419)
   - Format: A4 (210mm x 297mm)
   - Layout sections:
     1. Clinic header (logo, name, address, NIT, phone)
     2. Document title: "PLAN DE TRATAMIENTO"
     3. Patient information block (name, document, age, gender, city)
     4. Responsible doctor
     5. Date generated and plan creation date
     6. Diagnoses (CIE-10 codes and descriptions)
     7. Odontogram snapshot (PNG image, scaled to fit half-page width)
     8. Procedure table: | # | Diente | Codigo CUPS | Descripcion | Valor Estimado |
     9. Total estimated cost row (bold)
     10. Plan status and priority badge
     11. Approval section (if approved): patient signature image, signer name, signer document, approval date, IP
     12. Footer: page number, generation timestamp, legal disclaimer (firma digital)
8. Convert rendered HTML to PDF using WeasyPrint (or equivalent headless renderer). A4 page size, 15mm margins.
9. Upload generated PDF to S3: `tenants/{tenant_id}/pdfs/plans/{plan_id}/plan_{timestamp}.pdf`.
10. Store metadata in `plan_pdfs` table: plan_id, s3_key, generated_at, page_count, file_size_bytes, generated_by.
11. Write audit log entry (action: read (pdf_export), resource: treatment_plan, PHI: yes).
12. Generate pre-signed S3 URL with 1-hour TTL.
13. Return response per requested format (url or stream).

**Validation Rules:**

| Field | Rule | Error Message |
|-------|------|---------------|
| format | Must be one of: stream, url (if provided) | Formato no valido. Opciones: stream, url. |
| regenerate | Boolean (if provided) | El campo regenerate debe ser verdadero o falso. |

**Business Rules:**

- PDFs are cached in S3 and in the `plan_pdfs` table. A cached PDF is considered valid if it was generated after the plan's current `updated_at` timestamp. Any write to the plan (TP-04, TP-05, TP-06, TP-07, TP-08) invalidates the cached PDF.
- The odontogram snapshot is the current state of the patient's odontogram at the time of generation (not a historical snapshot). If the odontogram snapshot is unavailable, the section is omitted with a note "Odontograma no disponible."
- The approval signature section only appears in the PDF if `plan.approval_status = 'approved'`. Unapproved plans show "Pendiente de aprobacion del paciente."
- Branding elements (logo, colors) are fetched from the tenant's settings. If no logo is configured, the clinic name in text is used.
- `format = stream` is intended for direct print/download in the web UI; `format = url` is preferred for sharing via email or WhatsApp (TP-10).
- Cost values are formatted with locale-appropriate thousands separators and currency symbol (e.g., "$ 200.000" for COP).

**Edge Cases:**

| Scenario | Expected Behavior |
|----------|-------------------|
| Plan has 0 items | PDF generated with empty procedure table and note "No hay procedimientos registrados." |
| Clinic has no logo configured | Clinic name displayed as text in header; no broken image |
| Odontogram snapshot unavailable | Section shows "Odontograma no disponible al momento de la generacion." |
| Plan is draft (not yet approved) | PDF generated; approval section shows "Pendiente de aprobacion." |
| regenerate = true with existing cached PDF | Existing PDF replaced; new S3 key recorded |
| Long procedure list (> 30 items) | PDF spans multiple pages; pagination applied in procedure table |

## Side Effects

### Database Changes

**Tenant schema tables affected:**
- `plan_pdfs`: INSERT or UPDATE — PDF generation record (s3_key, generated_at, page_count, file_size_bytes, generated_by)

**Example query (SQLAlchemy):**
```python
pdf_record = PlanPdf(
    plan_id=plan_id,
    s3_key=s3_key,
    generated_at=datetime.utcnow(),
    page_count=page_count,
    file_size_bytes=file_size_bytes,
    generated_by=current_user.id,
)
session.merge(pdf_record)  # Upsert on plan_id
```

### Cache Operations

**Cache keys affected:**
- `tenant:{tenant_id}:patient:{patient_id}:treatment_plan:{plan_id}:pdf_url`: SET with 3600-second TTL (pre-signed URL cache)

**Cache TTL:** 3600 seconds (1 hour, matching S3 URL expiry)

**Jobs dispatched:** None (synchronous PDF generation; async generation for share is handled in TP-10)

## Performance

### Expected Response Time
- **Target:** < 1500ms (cache hit: pre-signed URL returned from DB)
- **Maximum acceptable:** < 5000ms (full PDF generation with odontogram snapshot)

### Caching Strategy
- **Strategy:** PDF stored in S3; pre-signed URL cached in Redis for 1 hour
- **Cache key:** `tenant:{tenant_id}:patient:{patient_id}:treatment_plan:{plan_id}:pdf_url`
- **TTL:** 3600 seconds
- **Invalidation:** On any write to treatment_plans or treatment_plan_items for this plan_id; on plan approval

### Database Performance

**Queries executed:** 5-7 (plan fetch, patient fetch, doctor fetch, tenant branding fetch, pdf_record upsert; plus odontogram + signature fetches if applicable)

**Indexes required:**
- `plan_pdfs.plan_id` — UNIQUE INDEX (one active PDF record per plan)
- `plan_pdfs.generated_at` — INDEX (staleness check)

**N+1 prevention:** All data fetched in parallel async queries before rendering.

### Pagination

**Pagination:** No (PDF is returned as a single document; internal page breaks handled by renderer)

| Input | Sanitization | Notes |

| patient_id | Pydantic UUID validator | Path parameter |
| plan_id | Pydantic UUID validator | Path parameter |
| format | Pydantic Enum validator | Query parameter |

### SQL Injection Prevention

**All queries use:** SQLAlchemy ORM with parameterized queries. No raw SQL.

### XSS Prevention

**Output encoding:** Jinja2 auto-escaping enabled for all template variables. No raw HTML injection.

### CSRF Protection

**Strategy:** JWT-based (stateless) — CSRF not applicable for API.

### Data Privacy (PHI)

**PHI fields in this endpoint:** All plan fields, patient name and document, doctor name, diagnoses, procedure descriptions, signature image, IP address in approval section.

**Audit requirement:** All PDF exports logged as PHI access events. Exported PDFs accessible only via time-limited pre-signed S3 URLs.

## Testing

### Test Cases

#### Happy Path
1. Generate PDF for approved plan (cache miss)
   - **Given:** Doctor, approved active plan with 3 items and signature
   - **When:** GET /pdf (format=url)
   - **Then:** 200 OK, pdf_url returned, from_cache = false, approval section in PDF, page_count >= 1

2. Return cached PDF URL
   - **Given:** PDF already generated after last plan update
   - **When:** GET /pdf (format=url, regenerate=false)
   - **Then:** 200 OK, pdf_url returned, from_cache = true, no regeneration

3. Force regeneration
   - **Given:** Cached PDF exists
   - **When:** GET /pdf with regenerate=true
   - **Then:** 200 OK, new PDF generated, from_cache = false

4. Return PDF stream directly
   - **Given:** Valid plan
   - **When:** GET /pdf with format=stream
   - **Then:** 200 OK, Content-Type: application/pdf, binary stream in body

5. PDF for draft (unapproved) plan
   - **Given:** Draft plan with items but no approval
   - **When:** GET /pdf
   - **Then:** 200 OK, PDF generated, approval section shows "Pendiente de aprobacion."

#### Edge Cases
1. Plan with 0 items
   - **Given:** Draft plan with no items
   - **When:** GET /pdf
   - **Then:** 200 OK, PDF with "No hay procedimientos registrados."

2. Clinic has no logo
   - **Given:** Tenant with no logo URL configured
   - **When:** GET /pdf
   - **Then:** 200 OK, PDF with text clinic name in header (no broken image)

3. Plan with 30+ items (multi-page)
   - **Given:** Plan with 35 items
   - **When:** GET /pdf
   - **Then:** 200 OK, page_count >= 2, all items present

#### Error Cases
1. Plan not found
   - **Given:** Non-existent plan_id
   - **When:** GET /pdf
   - **Then:** 404 Not Found

2. Rate limit exceeded
   - **Given:** 21 requests in 1 minute
   - **When:** 21st request
   - **Then:** 429 Too Many Requests

3. PDF rendering failure
   - **Given:** WeasyPrint raises exception
   - **When:** GET /pdf
   - **Then:** 500 pdf_generation_failed

### Test Data Requirements

**Users:** doctor, clinic_owner, assistant

**Patients/Entities:** Approved plan with signature and 3 items; draft plan with 0 items; plan with 35 items; tenant with logo; tenant without logo.
